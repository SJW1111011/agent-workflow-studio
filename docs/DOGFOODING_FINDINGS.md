# Dogfooding Findings - 2026-04-26

## Summary

通过真实使用 agent-workflow-studio 管理它自己的开发（T-801, T-802），我们完成了一个完整的人机协作循环，并发现了产品的根本性问题。

## 完成的协作循环

```
Agent 工作 (T-801: 添加骨架屏)
  ↓
Agent 记录证据 (workflow_done)
  ↓
人 Review (在 dashboard)
  ↓
人 Reject + 反馈 ("没有解决根本问题，需要真正优化")
  ↓
系统创建 Correction Task (T-802)
  ↓
Agent 收到反馈并继续工作 (T-802: 代码分割和动画优化)
  ↓
人 Review (第二轮)
  ↓
人指出：性能不是根本问题，根本问题是工作台和协作流程
```

**结论：核心协作循环工作了，但过程很痛苦。**

---

## 发现的根本性问题

### 1. 工作台的根本性问题

#### 问题：Dashboard 不是工作台，是监控面板

**现状：**
- ✅ 可以查看任务列表
- ✅ 可以查看任务详情
- ✅ 可以查看证据
- ✅ 可以 approve/reject
- ❌ **不能创建任务**
- ❌ **不能编辑任务**
- ❌ **不能记录完成**
- ❌ **所有操作都要回 CLI**

**痛点：**
- 想创建任务 → 回 CLI
- 想记录完成 → 回 CLI，输入长命令
- 想编辑任务 → 手动编辑 task.md
- 工作流程断裂，体验很差

**影响：**
- 用户不会使用产品
- 即使功能强大，体验太差
- 这是决定性的问题

**解决方案：** → T-803

#### 问题：交互模式不对

**现状：**
- 人需要手动编辑 task.md（文档工作，不是创意工作）
- 人需要记住复杂的 CLI 命令
- 人需要理解技术细节（MCP, orchestrator, claim...）

**应该是：**
- 人用自然语言描述任务
- AI 生成完整的 task.md
- 人 review 和调整（可选）
- 技术细节隐藏在幕后

**解决方案：** → T-805

#### 问题：信息架构混乱

**现状：**
- 任务列表、看板、详情...到处都是
- 不清楚重点在哪里
- 不清楚下一步该做什么
- 缺少清晰的工作流引导

**应该是：**
- 清晰的信息层次
- 明确的工作流引导
- 重点突出

**解决方案：** 需要单独的设计任务

### 2. 协作全流程的根本性问题

#### 问题：workflow_done 太复杂

**现状：**
```bash
npx agent-workflow done T-802 "优化性能" \
  --proof-path dashboard-next/vite.config.js \
  --proof-path dashboard-next/src/styles/app.css \
  --check "代码分割成功" \
  --check "bundle 减小 9%" \
  --check "动画优化" \
  --root .
```

**痛点：**
- 命令太长，参数太多
- 需要手动指定每个文件
- 需要手动写每个 check
- 容易出错，体验差

**应该是：**
```bash
npx agent-workflow done T-802 "优化性能"
→ 自动检测 git diff
→ 自动提取���动文件
→ 或者交互式选择
```

**或者：**
- 在 dashboard 一键完成
- 自动填充信息
- 人只需要 review

**解决方案：** → T-804

#### 问题：任务生命周期不清晰

**现状：**
- 什么时候可以 review？（我们修了很多 bug 才搞清楚）
- Agent 的职责是什么？（不清楚）
- 什么时候算"完成"？（不清楚）
- status: todo → in_progress → done，但实际使用时很混乱

**痛点：**
- 我们花了很多时间修 bug（approval flow）
- 但设计本身就不清晰
- 前后端不一致
- 数据结构不一致

**应该是：**
- 清晰定义任务生命周期
- 明确每个状态的含义
- 明确什么时候可以做什么操作
- 前后端一致

**解决方案：** 需要设计文档

#### 问题：反馈循环不完整

**现状：**
- Agent 完成工作后，不知道人什么时候 review
- Agent 不知道 approve 还是 reject
- 如果 reject，agent 怎么收到反馈？（通过 correction task，但不直观）
- 没有通知机制

**应该是：**
- Agent 完成工作 → 请求 review
- 人收到通知 → review
- 人 approve/reject → agent 收到通知
- 循环闭合，实时反馈

**解决方案：** 需要设计通知机制

#### 问题：没有真正测试 agent 工作流

**现状：**
- 我（Claude）用 CLI 手动执行
- 不是真的 agent 会话
- 如果是 Codex，流程会一样吗？
- 我们不知道

**应该是：**
- 用 Codex 真实测试
- 用 orchestrator 真实测试
- 发现真实的 agent 痛点
- 不是人模拟 agent

**解决方案：** → T-806

## 修复的问题

### ✅ Approval flow 前后端不一致

**问题：**
- 前端：只有 done 的任务才显示 approve/reject 按钮
- 后端：只有 done 的任务才能 review
- 但 agent 工作时任务是 in_progress
- 导致无法 review

**修复：**
- 前端：有 runs 就显示按钮
- 后端：检查 runs 目录，有 runs 就可以 review
- 统一逻辑

**Commit:** `69e4228 - Fix approval flow: allow review of in-progress tasks with runs`

### ✅ 数据结构不一致

**问题：**
- task.json 里没有 runs 信息
- runs 在单独的目录里
- 验证逻辑需要读取文件系统

**修复：**
- 验证逻辑读取 runs 目录
- 检查是否有 run 文件

**影响：**
- 这是临时方案
- 更好的设计：task.json 应该包含 runs 计数

## 关键洞察

### 1. Dogfooding 非常有价值

**通过真实使用，我们发现了：**
- 功能实现了，但不够健壮
- 用户体验很差
- 根本性问题比技术问题更重要

**如果没有 dogfooding：**
- 我们会继续优化性能
- 我们会继续添加功能
- 我们不会发现真正的问题

### 2. 产品方向是对的

**核心协作循环工作了：**
- Agent 工作 → 记录证据 → 人 review → 反馈 → agent 继续
- 这证明了产品的核心价值

**但执行不够好：**
- 过程很痛苦
- 需要修复很多 bug
- 用户体验差

### 3. 应该优先解决根本问题

**不是：**
- 优化性能
- 添加功能
- 美化界面

**而是：**
- 让工作台真正可用
- 让协作流程真正流畅
- 让用户体验真正好
