# Dashboard v2 - 重写决策和成果

## 决策：为什么重写？

### 触发点

在实现 T-803（Dashboard 操作能力）时，发现：
1. Modal 实现有 bug（页面卡死）
2. 代码审查发现严重的架构问题
3. 用户说："你有足够的权力，做出正确的选择，我不在乎这会多难"

### 旧 Dashboard 的问题

**代码审查发现的根本性问题：**

1. **状态管���过于复杂**
   - `useDashboardState.js`: 694 行
   - 单一 hook 承担太多职责
   - 状态逻辑、副作用、API 调用全部混在一起

2. **组件职责不清晰**
   - TaskList 包含 3 种视图 + 过滤 + 选择 + localStorage
   - 太多职责混在一起

3. **样式组织混乱**
   - `app.css`: 1400+ 行
   - 所有样式在一个文件
   - 容易冲突，难以维护

4. **缺少性能优化**
   - 没有 memo, useMemo, useCallback
   - 所有组件在任何状态变化时都重新渲染

5. **数据流不清晰**
   - 所有组件都能访问所有状态
   - 不清楚谁在修改什么

6. **用户体验差**
   - 信息过载
   - 交互不直观
   - 加载慢

**总代码量：2000+ 行**

### 为什么重写比重构更好？

**重构的问题：**
- 需要理解所有现有代码（694 行的 hook）
- 需要保持向后兼容
- 需要小心翼翼，怕破坏东西
- 可能最后还是要重写
- 时间更长，风险更高

**重写的优势：**
- 从零开始，正确的架构
- 只实现需要的功能
- 不需要向后兼容
- 代码清晰，易于维护
- 基于 dogfooding 的经验设计
- **实际上更快**

## Dashboard v2 架构

### 设计原则

1. **简单优于复杂**
   - 每个 hook 只做一件事
   - 每个组件只做一件事
   - 清晰的职责分离

2. **清晰的数据流**
   - Props down, events up
   - 不用复杂的 Context
   - 易于追踪和调试

3. **模块化**
   - CSS Modules（每个组件独立样式）
   - 清晰的文件组织
   - 易于���护和扩展

4. **只实现需要的功能**
   - MVP：查看、创建、详情、approve/reject
   - 不实现：复杂视图、复杂过滤、统计图表
   - 按需添加

### Hooks 设计

**每个 hook 只做一件事：**

```javascript
// useTasks() - 只管理任务列表
const { tasks, loading, error, refresh } = useTasks();

// useTaskDetail() - 只管理任务详情
const { task, loading, error } = useTaskDetail(taskId);

// useCreateTask() - 只管理创建任务
const { create, creating, error } = useCreateTask();

// useTaskReview() - 只管理 approve/reject
const { approve, reject, submitting, error } = useTaskReview(taskId);
```

**对比旧的：**
- 旧的：1 个 694 行的 hook 做所有事
- 新的：4 个小 hook，每个 30-40 行

### 组件设计

**清晰的职责分离：**

```jsx
Dashboard (主容器)
  ├─ Header (顶部栏 + 创建按钮)
  ├─ Sidebar (任务列表)
  ├─ Main (任务详情)
  └─ Modal (弹窗)
      └─ CreateTaskForm (创建表单)
```

**每个组件只做一件事：**
- `Modal`: 只负责弹窗逻辑
- `CreateTaskForm`: 只负责表单
- `Dashboard`: 只负责布局和状态协调

### 样式设计

**CSS Modules：**

```jsx
import styles from './Modal.module.css';

<div className={styles.overlay}>
  <div className={styles.content}>
    {children}
  </div>
</div>
```

**优点：**
- 样式局部作用域，不会冲突
- 每个组件的样式独立
- 易于维护

## 实现的功能

### MVP 功能（已完成）

1. **查看任务列表** ✅
   - 左侧边栏显示所有任务
   - 显示 ID、标题、优先级、状态
   - 点击选择任务

2. **查看任务详情** ✅
   - 右侧主区域显示详情
   - 显示元数据（优先级、状态、recipe）
   - 显示 runs 列表

3. **创建任务** ✅
   - 顶部 "+ Create Task" 按钮
   - Modal 弹窗
   - 表单：标题、优先级
   - 创建后自动跳转到新任务

4. **Approve/Reject** ✅
   - 任务详情页显示按钮（如果有 runs）
   - Approve：一键批准
   - Reject：输入反馈，创建 correction task

### 未实现的功能（按需添加）

- Kanban 视图
- Timeline 视图
- 复杂的过滤
- 统计图表
- 编辑任务
- 记录完成（workflow_done）

**原则：先把核心功能做对，再添加其他功能。**

## 关键决策

### 1. 不用 Context

**决策：**
- 不用复杂的 Context
- 用简单的 props 传递数据

**原因：**
- Context 适合主题、认证等全局状态
- 不适合频繁变化的业务数据
- Props 更清晰，易于追踪

### 2. CSS Modules

**决策：**
- 用 CSS Modules
- 不用单一的大 CSS 文件

**原因：**
- 样式局部作用域
- 不会冲突
- 易于维护

### 3. 只实现 MVP

**决策：**
- 只实现核心功能
- 不实现复杂视图、过滤、统计

**原因：**
- 先把核心功能做对
- 避免过度工程化
- 按需添加功能

### 4. 从零开始

**决策：**
- 不重构旧代码
- 从零开始重写

**原因：**
- 旧代码技术债太多
- 重构比重写更慢、更痛苦
- 重写可以正确的架构

## 经验教训

### 1. 不要在烂的基础上继续堆

**错误的做法：**
- 在 694 行的 hook 上继续添加功能
- 在混乱的组件上修 bug
- 在 1400 行的 CSS 上添加样式

**正确的做法：**
- 识别根本问题
- 做出正确的决策
- 从零开始，正确的架构

### 2. 重写比重构更快

**很多人认为：**
- 重写需要很长时间
- 重构更安全

**实际上：**
- 重写可以更快（2 小时 vs 可能几天的重构）
- 重写可以正确的架构
- 重构可能最后还是要重写

### 3. 简单优于复杂

**复杂的代码：**
- 694 行的 hook
- 20+ 个组件
- 复杂的 Context

**简单的代码：**
- 4 个小 hook
- 5 个组件
- 简单的 props

**简单的代码更好：**
- 易于理解
- 易于维护
- 易于扩展
- 性能更好

### 4. 用户体验 > 技术能力

**不重要：**
- 有多少功能
- 用了什么技术
- 代码有多复杂

**重要：**
- 用户能否完成任务
- 体验是否流畅
- 是否直观易用
